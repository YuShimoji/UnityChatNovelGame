「手がかりのロス（無駄）をなくす」こと、そして「一つの真実に収束するだけのリニア感を打破する」こと。この2つの非常に鋭い課題に対する解決策は、情報の扱い方を**「鍵（Key）」から「燃料（Fuel）」に変える**ことにあります。

従来のゲームでは、手がかりは「ドアを開けたら消滅する鍵」でした。 今回提案するのは、手がかりを**「組み合わせることで意味が変質し続ける化学物質」として扱う『推論合成システム（Inference Synthesis System）』**です。

コアコンセプト：『推論合成（Synthesis）』
プレイヤーの役割を、「情報を集めてパズルの穴を埋める作業員」から、「集めた断片を組み合わせて独自の仮説を組み立てる捜査官」へと変えます。

1. 「手がかりロス」の解決策：情報の多層化（Multi-layered Clues）
一つの手がかり（トピック）に、複数の「用途」を持たせます。ある謎が解けても、その手がかりは別の文脈で武器として残ります。

仕様： 1つのトピックに「事実（Fact）」と「文脈（Context）」の2つのパラメータを持たせる。

具体例：トピック「深夜の着信履歴」

用途A（事実）： アリバイ崩しパズルの解決に使用。（解決後も消滅しない）

用途B（文脈）： 浮気の証拠として、恋人を問い詰めるために使用。

用途C（文脈）： ストーカー被害の証拠として、警察（別キャラ）を動かすために使用。

結果： たとえパズルが解けたとしても、「この履歴、誰に見せるのが一番効果的か？」という戦略性が残り、死にステータス（ゴミ情報）になりません。

2. 「リニア化」の解決策：結論の選択（Hypothesis Building）
「ゲームが決めた真実に辿り着く」のではなく、**「プレイヤーが集めた手がかりの組み合わせによって、導き出される結論（エンディング）が変化する」**システムです。

仕様： 最後に自動的にエンディングが流れるのではなく、プレイヤーが**「重要だと思う手がかりを3つ選んで提出（告発）」**するフェーズを作る。

結果： 同じ「犯人はB」という結論でも、「証拠1・2・3」で告発した場合と、「証拠1・4・5」で告発した場合で、その後のBの反応（自白の内容や動機）や、周囲の運命が変わります。

実装すべきコアシステム仕様
この設計を実現するために、Yarn SpinnerとUnityで構築すべきシステムの具体像です。

A. 「推論ボード」UIの実装（Mental Board）
チャット画面とは別に、入手したトピック（アイコン）が並ぶ画面を用意します。

機能：Drag & Dropによる合成

トピックA「深夜の着信」＋ トピックB「脅迫メール」を重ねる

→ 新トピックC「ストーカー説」が生成される。

効果：

単に誰かに聞くだけでなく、「プレイヤー自身が脳内で考えたこと」をゲーム内で可視化させます。

合成して初めて「この件について聞く」という選択肢が出現します。

B. トピックの「属性」管理
Yarn Spinnerの変数管理だけでなく、ScriptableObjectでトピックを定義します。

C#

public class TopicData : ScriptableObject {
    public string id;
    public string displayName;
    // このトピックが持つ属性（タグ）
    public bool isPhysicalEvidence; // 物証
    public bool isEmotionalLeverage; // 感情的なゆさぶり（脅しなど）に使えるか
    public string[] relatedCharacterIDs; // 誰に関連するか
}
C. 「真実」の動的生成ロジック
これがリニア脱却の肝です。 ゲームの結末を「犯人はAだった」という1つの事実にするのではなく、**「プレイヤーがどの側面を暴いたか」によって結末の「色」**を変えます。

例：事件「横領」

ルートα（正義）： 物証だけで追い詰める → 犯人は逮捕されるが、動機は語られず、盗まれた金も戻らない。（ドライな結末）

ルートβ（情）： 犯人の「娘の病気（トピック）」を突きつけて説得する → 犯人は自首し、金も戻り、情状酌量される。（ウェットな結末）

ルートγ（共犯）： 情報を隠蔽し、犯人を脅して分け前をもらう。（悪の結末）

これらはすべて同じ「横領の証拠」というトピックから派生しますが、**「他にどのトピック（娘の病気など）を持っていたか」と「誰にどう使ったか」**で分岐します。

プレイヤー体験のフロー（UX）
このシステムにおける、プレイヤーの思考の流れです。

収集（Collect）： チャットや画像解析で「断片」を集める。

合成（Synthesize）： 「この着信と、あの証言は矛盾してるな…」と考え、推論ボード上で2つを合成する。

System: Topic_CallLog + Topic_Testimony = Topic_Contradiction (矛盾) を入手。

使用（Use）： 生成された「矛盾」を突きつける。

ここで「パズルとしての謎」は解ける。

再利用（Reuse）： しかし手元には「着信履歴」がまだ残っている。

「待てよ、この着信履歴、矛盾の証明には使ったけど、時間の不自然さを指摘すれば別の事実が出るのでは？」

プレイヤーは同じトピックを別の視点（合成相手を変える）で再利用しようと試みる。

PMからの推奨設計まとめ
ご希望の「ロスをなくし、全方向に伸びる」を実現するためには、以下の構成を仕様書の核心に据えてください。

情報は消費されない: 一度使ったトピックも消滅せず、ボードに残し続ける。

推論合成（Crafting）: トピック同士を組み合わせて「新しい質問事項」を生み出すメカニクスを入れる。（これがゲーム性の核になります）

告発の構成要素（Conclusion Builder）: エンディングは、最後にプレイヤーがボードから提出した「3つのトピック」の組み合わせによって計算される。
